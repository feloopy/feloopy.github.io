On large language models and decision science

Published: 2025-11-11 • Keivan Tafakkori

Introduction

Plainly put, I see large language models (LLMs), including the current generation of Generative AI, as systems meant to find logical tours/routes among characters, words, phrases, and larger linguistic patterns from a set of vocabularies/dictionaries/blogs/papers to generate text that fits a user’s prompt. There is nothing special about it.

LLMs can be thought of as input distance graphs for repeatedly solving a traveling salesman problem. Each sentence, paragraph, or article is a route through a set of nodes (characters, words, concepts), and the “distance” between nodes is learned or adjusted during training or fine-tuning.

Frequent co-occurrences in widely shared sources and recurring patterns pull some nodes closer together, so the model tends to favor those routes. That tendency explains why it leans toward familiar phrasings and can overproduce common sequences. When it follows less-traveled routes, "hallucination" becomes more likely.

Because of this mechanism, LLMs can follow formal logic, manipulate algebraic or code structures, and produce step-by-step instructions, demonstrating coherent chains of reasoning when the selected paths match valid patterns.

It is like a closed-book versus an open-book exam. I wrote about optimization and learning two years ago, and I believe this perspective remains valid in 2025 and beyond (refer to my LinkedIn post).

LLMs solve problems along paths they have learned. Before November 2022, people saved time and cost by outsourcing coding to experts in a field; now, they save time and cost by outsourcing prompting to experts in a field.

However, a small deviation from expectations from decisions suggested by LLMs can erode ROI and create real business losses. That is why, in many cases, the adoption of LLMs does not deliver a clear return on investment.

To conclude, there may already be one or two experts who can instantly respond to your request or who have documented how to respond to a prompt.

No expert can instantly solve a decision problem.

Decision problems require explicit preferences, context, and trade-offs.

That is Decision Science.

# knapsack_full.py
from typing import List, Tuple

def fractional_knapsack(items: List[Tuple[float, float, str]], capacity: float):
    """
    items: list of (weight, value, id)
    capacity: backpack capacity (float)
    Returns: (total_value, list_of_taken (id, taken_weight, taken_value))
    Greedy: takes highest value/weight ratio first.
    """
    # compute ratio and sort descending
    items_sorted = sorted(items, key=lambda it: (it[1]/it[0]) if it[0] > 0 else float('inf'), reverse=True)
    taken = []
    total_value = 0.0
    remaining = capacity

    for w, v, id_ in items_sorted:
        if remaining <= 0:
            break
        take_w = min(w, remaining)
        take_v = (v * (take_w / w)) if w > 0 else 0
        taken.append((id_, take_w, take_v))
        total_value += take_v
        remaining -= take_w

    return total_value, taken


def knapsack_01_dp(items: List[Tuple[int, int, str]], capacity: int):
    """
    0/1 knapsack DP.
    items: list of (weight:int, value:int, id)
    capacity: int
    Returns: (max_value, list_of_taken ids)
    Complexity: O(n * capacity) time and O(n * capacity) memory (we use optimized memory but still reconstruct choices).
    """
    n = len(items)
    # dp[w] = max value achievable with capacity w
    dp = [0] * (capacity + 1)
    # parent store: for reconstruction, store index of item used to reach dp[w] or -1
    parent = [[False] * (capacity + 1) for _ in range(n)]  # parent[i][w] = True if item i taken in optimal for capacity w after processing i

    for i, (wt, val, _) in enumerate(items):
        # iterate w from high to low to avoid reuse
        for w in range(capacity, wt - 1, -1):
            if dp[w - wt] + val > dp[w]:
                dp[w] = dp[w - wt] + val
                parent[i][w] = True

    max_value = dp[capacity]

    # reconstruct chosen items
    chosen = []
    w = capacity
    for i in range(n - 1, -1, -1):
        if parent[i][w]:
            wt, val, id_ = items[i]
            chosen.append(id_)
            w -= wt

    chosen.reverse()
    return max_value, chosen

def build_knapsack_model(m):
    # create a binary decision var for each item
    x_vars = []
    for name, w, v in items:
        # m.bvar creates a binary variable (per FelooPy examples)
        x = m.bvar(name=name)

Here it reaches a value of 429 — 38.96% better than the solution a non-expert obtained with ChatGPT. So yes, an LLM can act as an interface between user and solver, but it’s more like navigating a crowded road than walking an empty one: you save coding time but spend more time debugging. If you don’t know how to fix issues, even experts can spend much longer debugging. The code it produced works, but it could be written more cleanly as follows:

import feloopy as flp

n = 10
id = [f"item_{i+1}" for i in range(n)]
weight = [4,9,5,4,3,19,2,3,8,5]
value = [93,36,14,91,10,59,8,32,69,63]
capacity = 50

def knapsack(m):
    x = m.bvar(name="x", dim=[n])
    m.con(m.sum(weight[i]*x[i] for i in range(n))<=50)
    m.obj(m.sum(value[i]*x[i] for i in range(n)))
    return m

problem = {
    "environment":knapsack,
    "interface": "pulp",
    "solver": "cbc",
    "directions": ["max"]
}

model = flp.search(**problem)
x = model.get("x")

for i in range(n):
    if x[i]==1:
        print(id[i], "is selected")

Knowing the right tips and tricks can help you code better, build stronger models, and find better solutions. How much you improve depends on how much time you invest in learning. It’s not just about writing constraints in the correct mathematical form—each constraint is simple math. The real challenge is how constraints interact, and diagnosing those interactions is difficult even for experts.

Conclusion

Getting stakeholders to implement your recommendations—especially when they’ve relied on intuition for years—is one of the most cumbersome tasks for industrial engineers. When you write a research proposal or prepare a model/algorithm for deployment, you must clearly demonstrate business value to both the paper reviewer and the industry decision-maker; without that, your work risks rejection.

LLMs can deliver real value for routine, repetitive tasks. In operations research—and any field that automates repetitive work—code, models, and algorithms are tools, not sources of competitive advantage. The real edge comes from continually clarifying and updating the assumptions that define the problem so solutions genuinely fit real-world expectations or address system weaknesses. That’s continuous improvement: be context- and result-oriented, not tool-oriented.

We must also ask who builds systems, who makes decisions with LLMs, and who benefits. If an LLM only produces what an equally skilled developer could have written unaided, its added value is limited. Trust and expertise matter: a practitioner who understands the underlying theory and can actively collaborate with an LLM is far more reliable than someone who treats the process as a black box. Non-experts often remain uncertain about when to stop prompting or whether their prompts are adequate; experts, by contrast, can use LLMs to simplify repetitive tasks—but should avoid overreliance.
